<h2 id="新的日程表简介">新的日程表简介</h2>
<p>(来自 v0.1.0 发行概要，原先内容作废)</p>
<p>日程表集合了原先的课表，同时我加上了考试信息和物理实验信息。为了更加方便显示，只能引入午休和晚饭时间，并将一天结束时间定在21:25。另外，很久之前糊上去的数组也没了，改成了类似<a class="link" href="https://pub.dev/packages/calendar_view"  target="_blank" rel="noopener"
    >flutter_calendar_view</a>这样，按照连续时间来创建日程卡片。如果这个功能可以彻底和 PDA 解耦合，并将其用于其他的项目，这也许将是能够击败任何闭源课表的无敌存在(bushi)</p>
<p>这个实际上是我最开始写 PDA 的动机，很简单，一个页面显示所有跟我们学业相关的所有信息，不香吗？之前那个程序早期版本是彻底割裂的，后来也是仅仅在主页上搞了仨按钮，一个课表、一个考试、一个物理实验。其中后俩还是在明天/接下来有状况的状况下才有，总觉得有种割裂的美。</p>
<h3 id="课程信息数据模型介绍">课程信息数据模型介绍</h3>

</span></span></code></pre></div><h4 id="时间安排">时间安排</h4>
<p>包括以下部分：</p>
<ul>
<li>课程索引：课程信息在课程信息数组中的位置。下面我将介绍课程信息数组。</li>
<li>数据来源：在 PDA 中，有教务系统获取的课程，有用户添加的课程。这两类信息分别是保存在两个不同的课程信息数组的，这里用一个枚举类型<code>Source</code>变量来表示区别。</li>
<li>上课周次：一个布尔变量的数组，长度一定程度上代表了该学期的长度。从第一周开始，该周有课是<code>true</code>，否则是<code>false</code>。</li>
<li>星期几上课，第几节上课，第几节下课：分别使用一个<code>int</code>变量表示。请注意这里是将一天分成十节课来处理的，课程时间参见文件。星期几上课根据学校后端返回数据，从 1 开始，代表星期一。</li>
<li>可选教师信息：这个时间安排应该由谁上课，由于涉及到调课改变老师的因素，故在这里储存。</li>
<li>可选教室信息：上课地点信息。</li>
</ul>
<p>这里解释上课周次中“一定程度”的意义。实际使用中，大多数数据确实能符合这个情况，但很显然，会有很多的例外。我有个活跃用户，他有个课叫金工实习，那个课程的上课周次数组元素比其他的多 5 个，导致最后课表有很多的空白页面。这里讲道理，很无语，但暂时没有好办法解决。</p>
<p>另外有一个引申变量：</p>
<ul>
<li>上课长度：下课时间减去上课时间的长度，用于渲染课程表。</li>
</ul>

</span></span></code></pre></div><h4 id="课程调整信息">课程调整信息</h4>
<p>最复杂的类型，由于涉及到很多学校后端的东西，所以进行了很多注释，用来表示与其的对应关系。同时，我对后端数据没有进行过多处理，大多数进一步分析都是在该类里面处理的。</p>
<p>包括以下元素：</p>
<ul>
<li>调课类型：包括调课，停课，补课。这个将以一个枚举类型<code>ChangeType</code>来展示。</li>
<li>课程号，班级号，课程名：受影响课程的名称等信息。</li>
<li>原来周次信息和新的周次信息：都是一个可空的布尔数组，从第一周开始，如果有影响则为<code>true</code>，否则为<code>false</code>。</li>
<li>原先的老师和新的老师：为可空字符串，涉及到判断老师是否改变。</li>
<li>原先的课次信息和新的课次信息：一个只有两个元素的<code>int</code>数组，表示开始上课时间和停止下课时间。如果没有提供，默认给<code>[-1,-1]</code>。</li>
<li>原先的周次信息和新的周次信息：为可空的<code>int</code>变量，表示是否更改了上课时间。</li>
<li>原先的位置信息和新的位置信息：为可空的字符串变量，表示是否更改了上课地点。</li>
</ul>
<p>附带有这些方法：</p>
<ul>
<li>获取原先调整的周次和这次调整的周次信息：提取原来周次信息和新的周次信息数组中<code>true</code>的下标，方便后续处理。</li>
<li>判断老师是否改了：从后端获取的前后老师信息对比，看出老师信息改了没。这里涉及到一定的正则表达式和学校后端返回的老师信息，所以本程序里面是以我们学校的返回格式编写的。学校返回的老师信息使用半角逗号和空格<code>, </code>分隔，老师信息包含姓名和工号，中间用反斜杠<code>/</code>分隔。所以我这里，先去掉空格，然后用逗号和反斜杠来分隔，最后去掉工号，排序保证顺序一致。由此对两个数组进行处理，论证处理后的数据是否一致来判断是否更改老师。</li>
<li>显示给用户的原先老师和新老师信息：根据上面提供的信息，去掉数字和斜杠就好。</li>
<li>一个转换 ChangeType 为字符串的玩意。</li>
</ul>
<p>其中，调课类型枚组<code>ChangeType</code>如下：</p>

</span></span></span><span class="line"><span class="cl"><span class="c1"></span><span class="p">}</span>
</span></span></code></pre></div><h4 id="总体信息">总体信息</h4>
<p>在理想中，这将是传入课程表页面的东西。但在 PDA 中，由于添加删除用户自定义课程功能，导致必须引入程序的控制器，导致这个想法暂时没有实现:P</p>

<li>开学日期和当前学期代码，用来生成每周的索引和对应课程等。</li>
<li>两个课程信息数组。一个是教务系统发给的，一个是用户自行输入的。再次注意，里面有</li>
<li>两个时间安排数组，再次注意，里面有标识课程信息的类型和课程信息索引。</li>
<li>未排课数组，主要为了展示。</li>
<li>课程改变信息数组，用来进行调课处理。</li>
</ul>
<p>还有两个构造方法：</p>
<ul>

</ul>

<h3 id="一天的区分">一天的区分</h3>
<p>和前身课程表不同，本表格由于添加了午休和晚饭时间，所以需要说明一天的划分状况：</p>
<ul>
<li>早晨的 1-4 节课，每节课可以细分成 5 个小块来处理</li>
<li>午休按照 3 块来处理，表示 12:00 - 14:00</li>
<li>下午的 5-8 节课，每节课可以细分成 5 个小块来处理</li>
<li>晚饭时间按照 3 块来处理，表示 17:30 - 19:00</li>
<li>晚上的 9-11 节课，每节课可以细分成 5 个小块来处理</li>
</ul>
<p>很显然，这每个小块所代表的时长是不一样的，这是一种设计上的妥协。在实际计算中，如果不是按照课程那种“比较规整”的角度来决定开始结束的，那就需要计算时间在格子上的相对位置了。</p>
<h3 id="日程的计算与展示">日程的计算与展示</h3>
<p>首先，我们要知道<code>ClassOrgainzedData</code>是个啥。这是一个用来计算在一天之内，如何渲染卡片的类。它有六个属性：</p>
<ol>
<li>开始的位置和结束的位置，分别计为<code>start</code>和<code>stop</code>。</li>
<li>要显示的日程名称<code>name</code>和位置<code>place</code>。</li>
<li>卡片要显示的颜色<code>color</code>。</li>
<li>这段日程之内，有多少同时进行的，计为<code>data</code>。这是个动态类型的List，设计中只允许<code>TimeArrangement</code>,<code>ExperimentData</code>,<code>Subject</code>(考试信息)。</li>
</ol>
<p>具体如何计算开始和结束的位置，请结合“一天的区分”章节来看针对各个状况写的构造函数。</p>
<p>现在我们考虑一天之内，日程是如何展示的。以下的逻辑是在<code>/lib/page/classtable/class_table_view</code>里面，函数<code>List&lt;Widget&gt; classSubRow(int index)</code>过程的展现：</p>
<ol>
<li>如果传入的索引是 0，进行对最左边时间划分信息轴的渲染。否则进行下面的步骤。</li>
<li>从课程表信息中的时间信息<code>timeArrangement</code>，考试信息<code>subject</code>，物理实验信息<code>experiments</code>中，获取对应周次日期的日程信息。</li>
<li>按照开始时间排序。</li>
<li>按照<code>_checkIsOverLapping</code>分析可能的重复情况，如果有重复，按照重复的所有日程中最早的开始和最晚的结束，定义新的日程信息，其中包括所有这段时间里面经历的信息。这里应该能保证，最终形成的日程不会有任何交集。</li>
<li>根据生成的日程安排信息，使用<code>Positioned</code>组件，直接定位对应的位置，构建课程卡片。其中顶部和高度还要考虑到手机状态或平板状态，在<code>blockheight</code>中定义；左侧宽度由对应日期在一周中的位置确定。</li>
</ol>
<p>最终，这些东西一起扔到一个<code>Stack</code>里面，加上最上面显示日期周次信息的<code>ClassTableDateRow</code>，完成对日程表的渲染。<code>ClassTableDateRow</code>代码在<code>classtable_date_row.dart</code>，有三处值得注意：</p>
<ol>
<li>需要计算那一周周一的日期。</li>
<li>今天的颜色需要不一样。</li>
<li>长宽比不同的时候，字体的颜色不同。</li>
</ol>
<p>以上代码大幅度参考<code>flutter_calendar_view</code>插件。</p>

<h3 id="输出课表到-icalendar">输出课表到 iCalendar</h3>
<p>先从课程表页面状态读取生成的 iCalendar 字符串，然后进行分享。</p>
<p>生成该文件内容的代码在称为<code>iCalendar</code>的 getter 里面，以下是一些不正经介绍。</p>
<p>Initally I want to output classtable schedule to the system calendar, but it isn’t good. So far, it outputs the class schedule one by one, from the first class of the first week to the last class of the last week. I have to let users agree serveral times to import all class schedules, so ummm…</p>
<p>So I use iCalendar, a standard to transfer schedules. It can transfer the name of your schedule, the time range (start time and end time), and lots of additional infos, including email, alert, personnel etc.</p>
<p>For the class schedules, aka classtable, we only care about the place, time, and the name. According to the <a class="link" href="https://blog.woooo.tech/posts/about_linux_desktop/"  target="_blank" rel="noopener"
    ><strong>CYBER GOD OF OUR SCHOOL</strong></a>, we do not need a &ldquo;iCalender parser library&rdquo;, just treat it as a plain text file with mime-type <code>text/calendar</code>.</p>
<p>I will introduce a very simple iCalendar file here, and of course, it isn&rsquo;t cover all about iCalendar.</p>
<p>The whole iCalendar file is covered by <code>BEGIN:VCALENDAR</code> and <code>END:VCALENDAR</code>. While every schedule is covered by <code>BEGIN:VEVENT</code> and <code>END:VEVENT</code>. For each schedule, we can input the following attributes:</p>
<ul>
<li><code>SUMMARY</code>: A quickview of the schedule. In our case, the class’s name and place.</li>
<li><code>DESCRIPTION</code>: Detail description of the schedule. We can put teacher info in here.</li>
<li><code>DTSTART</code> and <code>DTEND</code>: The start time and the end time of the schedule. Notice we need to follow the time pattern <code>yyyyMMddTHHmmss</code>, a description is at <a class="link" href="https://icalendar.org/iCalendar-RFC-5545/3-3-5-date-time.html"  target="_blank" rel="noopener"
    >here</a> and <a class="link" href="https://pub.dev/documentation/intl/latest/intl/DateFormat-class.html"  target="_blank" rel="noopener"
    >here</a>.</li>
</ul>

<h2 id="关于物理实验乱码处理和-dio-转换器">关于物理实验，乱码处理和 Dio 转换器</h2>
<p>(来自 v0.4.0 发行概要，其中 Iconv 内容是本版本新添加的)</p>
<p>我们学校目前的物理实验服务器使用的是 2005 年的 ASP 技术，重点在 2005 年。实际上技术差点也没啥，但是有两点属实离谱：</p>
<ol>
<li>所有的信息都是用 GB2312 编码的。</li>
<li>传回的 Cookie 有中文字符的字段。</li>
</ol>
<p>其中第二点是最离谱的。</p>
<p>对于 Dart 底层的默认 UTF-16 String 来说，这俩点属实头疼。</p>
<h3 id="乱码处理">乱码处理</h3>
<p>乱码实际上很常见，常知道的锟斤拷梗就跟这个相关。毕竟汉字跟英文一样，在电脑底层都是需要用二进制编码来表示的。简体中文汉字有两个主要编码：</p>
<ol>
<li>国标码：一个用于编码汉字和一些日韩字符的国家标准，主要有 GB2312，GBK，GB18030 三个标准，呈现继承与发展（向下兼容）的特性。请查看<a class="link" href="https://zhuanlan.zhihu.com/p/453675608"  target="_blank" rel="noopener"
    >这个链接</a>来搞清国标码(GBK)相关。Windows 默认就是这个编码。国标码是定长编码，基本使用两个字节(16 位二进制位)来表示一个汉字。</li>
<li>UTF 编码：国际上有个统一码联盟，他们负责给全世界所有的字符编码，称为 Unicode。很早他们就支持了中日韩三个语言字符的编码（由于文字特性，中日韩字符在他们的体系中，在一个分区）。Unicode 只是规定了字符对应的二进制表示，但实际使用，位数过长而且浪费很多，所以实际使用只能继续缩短，使用更短的变长编码，称为 UTF。UTF 分成很多版本，一般代表了最短编码位数是多少。Linux / Mac + 互联网数据一般都是用这个编码。详情可以查看<a class="link" href="https://zhuanlan.zhihu.com/p/427488961"  target="_blank" rel="noopener"
    >这个链接</a>。</li>
</ol>
<p>说到变长编码知识，计算机组成会讲汇编命令是如何编码的，那里会讲的。</p>
<p>很明显，如果用 UTF 编码解析国标码，绝对会解析出不正常的数据。大巧不巧，Dart 语言的 String 本质上是一个 UTF-16 编码的序列。于是问题就产生了。</p>
<p>国标码是定长编码，而 UTF 是变长编码，很显然是基本没法兼容的。不兼容还好，在我的实践中，用 UTF 编码先编码回二进制信息，然后用国标码解码信息，大概率是无法得到正确的数据。</p>
<p>所以我目前程序中，需要让网络库不能用 Dart 的 String 来解码我的数据，我需要一个支持国标码的解码库。</p>
<h3 id="dart--flutter-的-gbk-解码库">Dart / Flutter 的 GBK 解码库</h3>
<p>这个实际上有两种：</p>
<ol>
<li>流行方案：使用 UTF 和 GBK 的码表一一对应，方便转换。这个方式对平台很灵活，缺点需要让我程序增大 500k 左右，而且这种方式在执行时候也会有些慢。</li>
<li>调用系统的解码接口来解码信息，我使用的是这个方案。但是缺点也很明显，如果没有对目标系统适配，解码就很难办。</li>
</ol>
<p>最终我使用的是这个库：<a class="link" href="https://pub.dev/packages/charset_converter"  target="_blank" rel="noopener"
    >charset_converter</a>。它目前能 Windows，Android，iOS 三个系统的转码，而且使用很方便。他支持很多编码，但我主要用国标码。</p>
<h3 id="关于-dio-的转换器">关于 Dio 的转换器</h3>
<p>Dio 的网络请求使用的是过滤器流水线模式：</p>
<blockquote>
<p>HTTP 请求 -&gt; 若干拦截器 -&gt; 转换器 -&gt; Dart 底层实现或系统网络实现
响应的二进制码 -&gt; 转换器 -&gt; 若干拦截器 -&gt; HTTP 响应</p>
</blockquote>
<p>拦截器一般处理 Cookie，判断响应码之类。目前 Dio 的拦截器不支持异步方法。</p>
<p>转换器 Transformer 是一个二进制码和 HTTP 请求响应结构互相转化的桥梁。默认的 Transformer 是解码后用来对 body 进行判断的。由于我上面提到，不能用 UTF 先编码再解码，所以我定制了一个 Transformer，称为 <code>ExperimentDioTransformer</code>。在一些基本对 Body 的二进制解析后，直接用 GBK 解码库来返回数据。学校物理实验服务器都是返回的网页，所以这么写没啥问题。</p>
<p>(现在新版本的拦截器可以用异步办法了，但我不想改了)</p>
<h3 id="关于-cookie-有中文字符">关于 Cookie 有中文字符</h3>
<p>可以看看<a class="link" href="https://github.com/cfug/dio/issues/1959"  target="_blank" rel="noopener"
    >我在 Dio 开发仓库提出的问题</a>。</p>
<p>Cookie 的官方规范，是仅允许一部分 ASCII 码作为合法字符的，Dart 核心库的 Cookie 实现严格遵照这个规范。但是令我哭笑不得的是，咱学校物理实验服务器传回的 Cookie 包含中文字符，就是这个用户的名字。加上 GBK 导致的编码，最后的结果自然就是报错，扔出“错误编码异常”。</p>
<p>人官方严格按照标准，无可厚非。我为了这个玩意折腾了很长时间，直到最后，有个人告诉我，那个 Cookie 给服务器传任何值都可以，我无语了&hellip;&hellip;</p>

<p>关于这里，想多说几句。<code>charset_converter</code>插件提供的接口本身没有提供待转换字符串的长度，这对其他原生平台没有问题，但对 Iconv 来说，有点要命。Iconv 接受的是一个用<code>\0</code>结束的，正宗 C 语言字符串，但是 Flutter 传过来的时候没有。而 C 语言没有侦测内存大小的函数，我的 C 语言基本也快忘了，所以情况很离谱。如果我不知道字符串长度，而直接转换，基本上不是末尾多了一个字符，就是少了几个，甚至直接报错。我直接改这个插件 Dart 端的函数形参很不现实，最主要的是这插件不是我写的。所以最后，我只能在 Dart 端先给字符串加个末尾的<code>\0</code>，然后传原生。</p>

<h3 id="如何登录">如何登录</h3>
<p>在统一认证服务认证用户后，需要有个跳转地址，这个地址根据我的抓包，应该是这个：</p>
<pre tabindex="0"><code>https://xxcapp.xidian.edu.cn/a_xidian/api/cas-login/indexredirect=https%3A%2F%2Fxxcapp.xidian.edu.cn%2Fuc%2Fapi%2Foauth%2Findex%3Fappid%3D200190304164516885%26redirect%3Dhttps%253A%252F%252Fehall.xidian.edu.cn%252Fjwmobile%252Fauth%252Findex%26state%3DSTATE%26qrcode%3D1&amp;from=wap
</code></pre><p>他很长，很显然用到了其他的二级鉴权服务，但是我们不需要关心，在最开始跟着他 302 的 Location 跳转就好了。为啥是“在最开始”，因为跳转的 Location 里面有很重要的鉴权标志。</p>
<p>接下来的内容涉及到 Authorization 头，建议各位先看个<a class="link" href="https://developer.mozilla.org/zh-CN/docs/Web/HTTP/Headers/Authorization"  target="_blank" rel="noopener"
    >这玩意的介绍</a>。我的程序中，有个双创招队伍展示的功能，那个的后端也是用的这玩意鉴权，而且显然比这玩意好写。</p>
<p>我上面提到了很多的 302 跳转，其中最重要的跳转是在 xxcapp 之后 get 的第一个 ehall ，那里面的 Location 充满了 token，那个 token 就是我们之后要用到的 Authorization token。</p>
<p>之后用功能的时候，需要在 Header 里面使用 Authorization，传过去这个 token 就行。</p>
<p>在使用一个服务的时候，根据抓包看出的规律，我们需要给 <a class="link" href="https://ehall.xidian.edu.cn/jwmobile/biz/home/updateServiceUsage"  target="_blank" rel="noopener"
    >https://ehall.xidian.edu.cn/jwmobile/biz/home/updateServiceUsage</a> POST 一个 json 编码的数据，形式是 {&ldquo;key&rdquo;: 你要搞的服务}。</p>
<p>其中你要搞的服务有这些：</p>
<div class="highlight"><pre tabindex="0" class="chroma"><code class="language-json" data-lang="json"><span class="line"><span class="cl"><span class="p">{</span>

<p>首先，安卓小部件我是求别人写的，苹果部件是我写的。(Swift UI 是真的好写)</p>
<h3 id="苹果的-groupid-机制">苹果的 groupid 机制</h3>
<p>关于数据读取，安卓的好办，直接读就行了。但是苹果的就难办了。</p>
<p>简单来说，苹果的小部件是独立于主程序的，而苹果的沙盒机制导致了程序之间的数据是无法共享的。各位从 App Store 上面下载的程序，本质上是一个套件，里面包含了主程序和小部件程序。虽然小部件程序不能直接读主程序的数据，但是苹果大发慈悲地提供了一种名字叫<code>groupid</code>的机制，来使一个套件下面的数据能够共享。对于这个东西的介绍，可以阅读<a class="link" href="https://www.jianshu.com/p/a2d9ef58bad5"  target="_blank" rel="noopener"
    >这个文章</a>。</p>
<p>我们需要共享的数据，主要是课表数据。所以我们需要把课表数据从主程序的区域复制一份到公共区域，而这里又涉及到调用原生端代码的问题，因为那些存储到公共区域的代码都是原生代码。这里我们需要提到<a class="link" href="https://pub.dev/packages/pigeon"  target="_blank" rel="noopener"
    >Flutter 的 Pigeon 插件</a>，通过这个插件，我们可以比较简单地访问到原生代码。具体使用这里不会说明，我主要介绍我在原生开放了啥接口。</p>
<p>这个接口需要获取一个<code>appid</code>组名称，一个文件名<code>fileName</code>，文件里面的内容<code>data</code>（必须为纯文本）。同时需要实现一个函数<code>saveToGroupId</code>来实现保存到公共空间的功能。接下来展示的就是我给插件编写的模板接口文件，插件会根据这个文件生成用于访问该函数的类。顺便一提，这个文件不能放进<code>/lib</code>里面。</p>

<h3 id="数据处理">数据处理</h3>
<p>这里处理方式双端大差不差：</p>
<ol>
<li>读取课程表文件，考试信息文件，物理实验文件。</li>
<li>从这些文件中提取出当日/明日信息，并转化为用于显示的日程类。</li>
<li>日程类渲染展示。</li>
</ol>
<p>此外，苹果部件在渲染前还包含一步，也就是创造小部件时间轴，根据时间删除过去的日程。这个功能安卓没实现，因为安卓后台刷新更加玄学。或者这么说，所有平台的后台刷新都是玄学级别，不如苹果小部件的时间轴可以控制。</p>
<h3 id="小部件刷新">小部件刷新</h3>
<p>安卓的是靠一个内置的时钟，每十八分钟更新。不过这玩意玄学能不能更新，只能看天。</p>
<p>苹果是按照时间轴机制刷新的，每天半夜根据文件内容，生成当天的时间轴，包括今天和明天的内容。关于时间轴请查看<a class="link" href="https://developer.apple.com/documentation/widgetkit/keeping-a-widget-up-to-date"  target="_blank" rel="noopener"
    >苹果的官方介绍</a>。我的处理方法是，先获取数据，整理后根据各个日程的下课时间来决定时间轴上面的坐标，对应坐标是当时还在进行的课程和尚未开始的课程，最后进行刷新。</p>
<p>在此之前，我打算使用一个后台刷新插件，名字叫 <a class="link" href="https://pub.dev/packages/background_fetch"  target="_blank" rel="noopener"
    >background_fetch</a>，通过这个东西打算在后台让程序运行，进而在主程序主页的日程组件更新日程的时候，顺带更新小部件的数据。但是后台刷新在任何平台都是玄学，加上开学后有个叫跑步的玩意彻底让体育小插件没有存在的意义，顺便带走了后台刷新的意义，于是就这样了()</p>

<h2 id="一些乱七八糟的玩意">一些乱七八糟的玩意</h2>
<h3 id="关于开源的想法">关于开源的想法</h3>
<p>我对软件，按照开源和开发者，这么看：</p>
<blockquote>
<p>个人开发的开源软件或半开源软件 &gt; 集体开发的开源软件 &gt; 个人开发的闭源软件 &gt; 集体开发的半开源软件 &gt; 集体开发的闭源软件</p>
</blockquote>
<p>其中，半开源软件请参考 <a class="link" href="https://f-droid.org/zh_Hans/docs/Anti-Features/"  target="_blank" rel="noopener"
    >FDroid 的负面特征定义</a>。显然我的软件属于半开源软件，我这个软件实质上模拟了你在浏览器中，对学校后端的访问。</p>
<p>实际上软件的开源与否，并不重要，重要的是软件本身能不能很好用，而按照我的经验，软件的好用也可以这么排序，尤其是手机端应用()</p>
<p>所以，我虽然经常说开源很重要，但这个实际上是因为我认为个人开发者的产品更好而导致的。而开源软件放前面，是因为代码开放让人用着更舒服，可能我长期用 Linux 留下来的某种遗留症状。而且我某种意义上，真的不喜欢封闭的东西，虽然我发现大家都喜欢。</p>
<p>而为啥我要将这个软件按照 MPL 授权，是因为我的软件有很多可以复用的东西，比如上面我大幅度提到的课程表和那个 BothSide 。这些复用的东西我将来是打算做成程序内的 package，如果按照 GPL ，不利于传播。而我目前程序状态，如果使用 MIT 之类的，那可能会有很多的魔改版，然后闭源了。MPL 是按照文件强制开源的，就目前状态所言，假如你只是用了我的课程表代码文件，那么，你只需要开源课程表代码文件+你对这个代码的修改，就好了。</p>
<h3 id="关于上架-ios">关于上架 iOS</h3>
<p>目前我打算这个版本尝试申请 Testflight。据我所知，至少有三个组+两个人也在写这个东西，我无论如何也得打出去第一炮。</p>
<blockquote>
<p>这里我无端想到了《东周列国春秋篇》电视剧里面的要离。</p>
<p>中学学过“专诸刺王僚，要离刺庆忌”，不知道咋回事。看了电视剧才知道，他为了出名，壮士断腕。吴王阖闾说：“你是要名，还是要家？”结果就不必说了……</p>
<p>我现在也有点那啥，我为了这玩意，已经砸进去很多了。我这辈子都没一次性花这么多钱，现在我不上架，真对不住那么钱了。但上架了话，真的会有那么多人用嘛？</p>
<p>我这玩意，真要跟电表，跟其他原生，可以说是被爆打。也许就真的只是“开源+第一个上架”？开源这年头算毛线的优势？</p>
<p>写这个程序有一段时间，我一直在想这件事，不过现在释然了。</p>
</blockquote>
<p>本来我是不想现在就上架 App Store，但是电表突然上架了。虽然目前功能少，但着实打了一惊，我也顾不上我软件的不成熟，也上架了。看来大家还是很认可我的软件，所以感觉可以。我也很感谢很多帮我的人，无论是画吉祥物的，还是帮我发传单的，给我 UI 设计提出建议的。</p>
<p>之前我好像说过学校“揭榜”的事情，这玩意确实有点用，就是在面试时候问项目背景的时候，至少能扯到学校:-P</p>
